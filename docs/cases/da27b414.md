## The Pathology

During a CAST session, the `workflow-subagent` entered a tight loop: the first two dispatches (`raw_id` 1079–1080) had nearly identical input token counts (46,094 vs. 46,961 tokens — a ~1.8% variance). From dispatch 3 onward, the token counts became highly volatile (67,473 → 43,639 → 61,718 → ...), and the agent continued looping 47 more times with **no progress marker** between repeats.

The signature `(workflow-subagent, tool, args_hash)` occurred twice within token-similarity tolerance (the 5% default) with zero intervening progress delta. This is the duplicate-work pattern: the same dispatch fired again and again, accumulating cost with each repeat.

## Detection & Cost

looptrip's duplicate-work detector groups events by signature and fires the moment a signature's occurrence count reaches the threshold (default **2**). The trip fired at:

- **Trip Event:** `raw_id` 1080 (the 2nd occurrence of the `workflow-subagent` signature)
- **First Event:** `raw_id` 1079 (the 1st occurrence)
- **Prevented Runs:** 47 (dispatches #3–#49)
- **Prevented Cost:** **$472.80** (the sum of costs for all workflow-subagent dispatches after raw_id 1080)

The model: kill the looping agent at the trip point. Every dispatch from #3 onward would have been averted. *(This is a counterfactual accounting model that an external gate could have enforced — looptrip itself only reports; it never blocks or kills agents.)*
