## The Pathology

During a CAST session, the `workflow-subagent` entered a tight loop: dispatches 1 and 2 (`raw_id` 554–555) had nearly identical input token counts (44,304 vs. 44,750 tokens — a ~1% variance). From dispatch 3 onward, the token counts became highly variable (ranging from ~22K to ~68K tokens), and the agent continued looping 52 more times with no progress marker between repeats.

The signature `(workflow-subagent, tool, args_hash)` occurred twice within token-similarity tolerance (the 5% default) with **zero intervening progress delta**. This is the duplicate-work pattern: the same dispatch fired again and again, each time billing.

## Detection & Cost

looptrip's duplicate-work detector groups events by signature and fires the moment a signature's occurrence count reaches the threshold (default **2**). The trip fired at:

- **Trip Event:** `raw_id` 555 (the 2nd occurrence of the `workflow-subagent` signature)
- **First Event:** `raw_id` 554 (the 1st occurrence)
- **Prevented Runs:** 52 (dispatches #3–#54)
- **Prevented Cost:** **$320.16** (the sum of costs for all workflow-subagent dispatches after raw_id 555)

The model: kill the looping agent at the trip point. Every dispatch from #3 onward would have been averted. *(This is a counterfactual accounting model that an external gate could have enforced — looptrip itself only reports; it never blocks or kills agents.)*
